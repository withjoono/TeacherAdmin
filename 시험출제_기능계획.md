# 시험 출제 · 자동채점 · 분석 · 학부모 통보 — 구현 계획

작성일: 2026-05-24
대상 앱: TeacherAdmin(출제) · ExamHub(시험·분석 엔진) · TutorBoard=Classboard(학생 풀이) · ParentAdmin(통보)
연동 방식: **ExamHub 기존 모의고사 시스템 확장** (사용자 선택)

---

## 1. 목표 기능

선생님이 TeacherAdmin에서 시험을 출제한다 — 총 몇 문항인지, 각 문항이 객관식·단답형·주관식 중 무엇인지, 배점과 정답을 정한다. 출제는 손이 덜 가도록 자동화한다(전체 객관식 일괄 지정, 전체 배점 일괄 지정 등). 학생은 Classboard(TutorBoard) 또는 ExamHub에서 담당 선생님을 골라 시험을 풀고, 자동채점·분석 결과를 받는다. 분석은 ExamHub가 이미 가진 과목선택·성적분석·성적추이·오답 화면을 재사용한다. 채점이 끝나면 학부모앱(ParentAdmin)에 결과를 통보한다.

---

## 2. 현황 진단 (실제 코드 확인 결과)

| 앱 | 확인된 자산 | 부족한 점 |
|---|---|---|
| **ExamHub** | `eh_mock_exams`·`eh_exam_questions`·`eh_student_answers` 문항단위 모의고사 + 자동채점(`/mock-exams/grade`, `/admin/grading/submit`) + 분석 모듈(`score`·`analysis`·`statistics`·`weakness-analysis`·`wrong-answer`) | `eh_exam_questions.answer`가 `Int`(객관식 1~5 전용) — **문항 유형·텍스트 정답·주관식 개념 없음**. `eh_mock_exams`에 **출제자(선생님)·대상 반·발행 상태 없음** |
| **TutorBoard (=Classboard)** | `src/examhub/`에 ExamHub 화면이 **이미 임베드**됨 — `ExamCategorySelector`(과목·시험 선택), `ScoreAnalysisPage`, `StatisticsPage`, `WeaknessAnalysisPage`, `WrongAnswersPage`. 백엔드 `tests` 모듈, `tb_tests`·`tb_notifications` 테이블 | "담당 선생님 시험"을 고르고 푸는 진입·풀이 화면 없음 |
| **TeacherAdmin** | `tutor` 모듈에 `tb_tests`(제목+점수만) 단순 시험. ExamHub mockExam API를 부르는 미완성 페이지(`exams`·`grading`) | 문항 구성·자동채점·출제 자동화 전무 |
| **ParentAdmin** | Next.js 프론트 + `tutor`·`school` 백엔드. ExamHub `Member`에 `parentPhone`·`parentEmail` 보유 | 시험 결과 통보 경로 미연결 |

요약하면, **문항단위 시험·자동채점·분석 엔진(ExamHub)과 학생측 분석 화면(TutorBoard 임베드)은 이미 있다.** 빠진 것은 ① 선생님이 자유 구성(객관식·단답형·주관식)하도록 하는 데이터 모델 확장, ② 출제 자동화 UI, ③ 학생 풀이 화면, ④ 학부모 통보다.

## 3. 아키텍처 — 데이터 흐름

```
[TeacherAdmin] 출제 마법사
      │  (eh_mock_exams · eh_exam_questions 생성)
      ▼
[ExamHub DB]  eh_mock_exams (owner=선생님)
      │
      ▼
[TutorBoard/ExamHub] 학생: 담당 선생님 → 시험 선택 → 풀이 → 제출
      │  (eh_student_answers 기록)
      ▼
[자동채점] 객관식·단답형 즉시 → 주관식은 '채점 대기'
      │
      ├─▶ [ExamHub 분석] ScoreAnalysis · WrongAnswers · Weakness · Statistics (재사용)
      └─▶ [학부모 통보] tb_notifications / Hub 알림 → ParentAdmin
```

정본 저장소는 ExamHub의 `eh_*` 스키마다. TutorBoard의 단순 `tb_tests`(제목+점수)는 이번 문항단위 시험과 별개이므로 그대로 두거나 점진 폐기한다.

## 4. 데이터 모델 확장 (ExamHub `prisma/schema.prisma`)

기존 모의고사 모델을 깨지 않고 **선택 필드를 추가**한다. 공식 모의고사(`category="official"`)는 영향 없음.

### 4-1. `MockExam` (eh_mock_exams) — 출제자·발행 정보 추가
- `ownerType` `String @default("official")` — `"official"` | `"teacher"`
- `ownerId` `String?` — 선생님 Hub member_id (teacher 출제 시)
- `targetClassId` `String?` — 대상 반(Mentoring.classId)
- `status` `String @default("published")` — `"draft"` | `"published"` | `"closed"`
- `totalQuestions` `Int?` · `dueDate` `DateTime?`

### 4-2. `ExamQuestion` (eh_exam_questions) — 문항 유형 추가
- `questionType` `String @default("objective")` — `"objective"`(객관식) | `"short"`(단답형) | `"essay"`(주관식)
- `answer` `Int?` — 객관식 정답(1~5). nullable로 완화
- `answerText` `String?` — 단답형 정답(여러 개 허용 시 구분자)
- `gradingMode` `String @default("auto")` — `"auto"` | `"manual"`(주관식)
- `choiceCount` `Int @default(5)` — 객관식 보기 수(4·5지선다 대응)

### 4-3. `StudentAnswer` (eh_student_answers) — 텍스트 답안·수동채점
- `selectedAnswer` `Int?` — 객관식 선택. nullable로 완화
- `answerText` `String?` — 단답형·주관식 작성 답안
- `isGraded` `Boolean @default(true)` — 주관식 채점 전 `false`
- `gradedBy` `String?` · `gradedAt` `DateTime?` — 수동채점 기록

### 4-4. 시험 요약 (선택)
수능형 `eh_student_scores`는 영역별 원점수 구조라 선생님 단원평가에 맞지 않는다. 선생님 시험은 `eh_student_answers` 합계로 충분하므로, 가벼운 요약 테이블 `eh_exam_submissions`(memberId, mockExamId, totalScore, earnedScore, correctCount, submittedAt, status) 신설을 권장한다.

> 마이그레이션 1건 필요. 모두 추가/nullable 완화라 기존 데이터 안전.

## 5. 출제 자동화 (TeacherAdmin) — "손이 덜 가는 출제"

출제는 **마법사(wizard)** 형태로, 기본값을 최대한 깔아두고 예외만 손보게 한다.

1. **1단계 · 기본 정보** — 시험 이름, 대상 반, 응시 기간.
2. **2단계 · 문항 수** — 총 문항 수 N 입력 → N개 문항이 자동 생성된다.
3. **3단계 · 일괄 기본값** — 한 번의 클릭으로 전체 적용:
   - `전체 객관식` / `전체 단답형` / `전체 주관식` — 모든 문항 유형 일괄 지정
   - `전체 배점 N점` — 모든 문항 동일 배점
   - `100점 균등 분배` — 100 ÷ N 자동 계산(나머지는 앞 문항에 +1)
   - `객관식 보기 수` 4지 / 5지 일괄
4. **4단계 · 예외 수정** — 표(grid)에서 문항별로 **달라지는 것만** 수정. 정답은 한 행씩, 또는 `정답 일괄 입력`(예: `3 1 4 2 5 ...` 공백 구분)으로 한 번에.
5. **5단계 · 검토 → 발행** — 미입력 정답·배점 합계 경고 후 `발행`(status=published).

핵심 원칙: **새 문항은 항상 직전 문항의 유형·배점·보기 수를 상속**한다. 25문항 전부 5지선다 3점이면 → 문항 수 25 + `전체 객관식` + `전체 배점 3점` + 정답 일괄 입력, 클릭 4번으로 출제 완료.

## 6. 단계별 구현 계획

### Phase 1 — 데이터 모델 확장
ExamHub `schema.prisma`에 4장의 필드 추가 + `eh_exam_submissions` 신설 → 마이그레이션. 기존 공식 모의고사 무영향 확인.

### Phase 2 — TeacherAdmin 출제 마법사
- 미완성 `exams`·`exams/new` 페이지를 5단계 마법사로 재구성(또는 신규 `exam-builder` 라우트).
- 백엔드: ExamHub의 `/admin/mock-exams` 계열 엔드포인트를 확장(문항 유형 파라미터)하거나, TeacherAdmin이 ExamHub DB에 크로스 스키마로 직접 쓰기(StudyPlanner 플래너 연동 때 쓴 패턴과 동일). **권장: ExamHub에 선생님 출제용 엔드포인트를 추가**하고 TeacherAdmin이 호출 — 채점 로직과 한곳에 모임.
- 출제자 식별: TeacherAdmin JWT의 Hub member_id → `MockExam.ownerId`.

### Phase 3 — 자동채점 + 주관식 수동채점
- 제출 시 채점: **객관식** `selectedAnswer == answer`; **단답형** 정규화(공백·대소문자 제거) 후 `answerText` 비교(복수 정답 허용); **주관식** `isGraded=false`로 보류.
- 주관식 수동채점 화면: TeacherAdmin에 "채점 대기" 목록 → 학생 답안 보고 점수 부여 → `eh_student_answers` 갱신 → 합계 재계산.

### Phase 4 — 학생 풀이 화면 (Classboard=TutorBoard, ExamHub)
- TutorBoard `src/examhub/`에 "담당 선생님 시험" 진입점 추가 → 담당 선생님별 시험 목록 → 풀이 화면.
- 풀이 화면은 문항 유형별 입력 UI: 객관식(보기 버튼), 단답형(짧은 입력), 주관식(텍스트영역).
- 제출 → 자동채점 → 즉시 결과(객관식·단답형) + 주관식 "채점 중" 표시.
- ExamHub 본체에도 동일 진입을 두어 "Classboard 또는 ExamHub" 양쪽 충족.

### Phase 5 — 분석 화면 연결 (ExamHub 자산 재사용)
- TutorBoard에 이미 임베드된 `ScoreAnalysisPage`·`WrongAnswersPage`·`WeaknessAnalysisPage`·`StatisticsPage`는 `eh_student_answers` 기반이므로 **선생님 시험 결과가 자동 반영**된다.
- `ExamCategorySelector`에 "담당 선생님 시험" 카테고리를 추가해 공식 모의고사와 함께 고르게 한다.
- 수능형 표준점수·백분위(`eh_student_scores`) 의존 화면은 선생님 시험에선 원점수·정답률·오답 중심으로 표시하도록 분기.

### Phase 6 — 학부모 통보 (ParentAdmin)
- 채점 완료(주관식 포함 전체) 시점에 알림 생성: TutorBoard `tb_notifications` 레코드 + Hub 공유 알림.
- ParentAdmin / TutorBoard `ParentDashboard`가 해당 알림과 결과 요약(총점·정답수·오답 링크)을 노출.
- `Member.parentPhone`·`parentEmail` 보유 → 문자/메일 발송은 선택 옵션(2차).

## 7. 점검·결정 필요 사항

- **식별자 정합성** — TeacherAdmin·TutorBoard·ExamHub·ParentAdmin이 학생/선생님을 Hub member_id로 일관되게 매핑하는지 확인(StudyPlanner 연동 때와 동일 이슈).
- **연동 방식** — ExamHub에 엔드포인트 추가(권장, 결합도 낮음) vs TeacherAdmin이 ExamHub 스키마 직접 접근(작업량 적음). 채점 로직 일원화를 위해 전자 권장.
- **수능형 분석과의 분리** — `eh_student_scores`(영역별)는 선생님 시험에 부적합 → 선생님 시험은 `eh_student_answers` + `eh_exam_submissions` 요약만 사용하고, 수능 특화 분석(표준점수·정시예측)은 비활성.
- **단답형 채점 관용도** — 띄어쓰기·동의어·복수 정답 처리 규칙을 출제 시 정할 수 있게 할지.
- **TutorBoard `tb_tests`와의 관계** — 제목+점수만 있는 단순 시험은 유지할지 폐기할지.

---

## 8. 권장 진행 순서 요약

Phase 1(모델 확장)이 모든 것의 전제다. 이어 Phase 2(출제 마법사)로 선생님이 시험을 만들 수 있게 하고, Phase 3(자동채점)·Phase 4(학생 풀이)로 핵심 루프를 완성한 뒤, Phase 5(분석 연결)는 대부분 재사용이라 작업량이 적고, Phase 6(학부모 통보)으로 마무리한다. 출제 자동화(5장)는 Phase 2의 핵심 UX 요구사항으로 함께 설계한다.
